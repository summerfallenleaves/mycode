### 核心思想

**Agent Core 只写一遍，驱动所有前端。** 核心是一个纯 TypeScript 库，不依赖任何 UI 框架。它通过 `AsyncGenerator<AgentEvent>` 对外发布标准事件流，任何前端（终端 Ink、Web React、第三方 SDK）都可以订阅和渲染。

## 二、核心设计原则

### 1. 向外订阅，向内无引用

Agent Core 永远不 import React、Ink、DOM 或任何 HTTP 库。它只负责：
- 调用 LLM 获取推理和工具调用
- 执行工具并收集结果
- 产生标准的类型化事件流

### 2. AsyncGenerator 是唯一的事件通道

所有实时输出（LLM 文本流、工具调用、状态变更）都通过同一个 `AsyncGenerator` 发出。这意味着：
- CLI 可以直接 `for await...of` 消费
- Web 可以将 Generator 映射为 SSE
- 无需 WebSocket 即可实现双向流

### 3. 工具执行是插件化的

工具（Tool）分两类：
- **内置工具**：直接以 TypeScript 函数定义（读写文件、执行命令、搜索代码）
- **MCP 工具**：通过 `@modelcontextprotocol/sdk` 连接外部 MCP 服务器

两者在 Agent Core 层面是同一接口，统一注册、统一调用、统一超时/错误处理。

### 4. 会话管理独立

Agent 的上下文窗口、历史压缩、Token 计数由 Core 管理，不泄漏到 UI 层。UI 只关心展示当前事件，不关心历史如何压缩。

### 关键选型表

### 选型说明

#### 为什么选 Vercel AI SDK 而不是 LangChain.js

| 对比维度 | Vercel AI SDK | LangChain.js |
|----------|--------------|--------------|
| 流式 tool call | ✅ `maxSteps` + `fullStream` 原生支持 | ⚠️ 需要自行组合 callback |
| 包体积 | 轻量，tree-shakable | 较重，大量抽象层 |
| 版本迭代 | 活跃（v6，~2小时一发） | 稳定但较慢 |
| 与 Next.js 集成 | 一等公民 | 需适配层 |
| 社区趋势 (2026) | 主流（10.5M npm/周） | 用户基数仍在但增长放缓 |

**结论**：Agent Core 的 LLM 抽象基于 Vercel AI SDK，但通过一层薄薄的 adapter 封装，避免直接耦合到具体 SDK。

#### 为什么保留 Hono 作为可选组件

- **WebSocket 场景**：如果 Agent 需要实时推送（如日志流、后台任务状态），Hono 的原生 WebSocket 支持比 Next.js API Routes 更轻量
- **MCP 服务器**：Hono 是构建 MCP 服务器的理想选择（<20KB，可部署到 Cloudflare Workers）
- **独立部署**：Hono 服务可以独立于 Next.js 部署，作为 Agent 的 API 网关

#### 为什么不选 LangGraph.js

LangGraph.js 适合复杂的图状工作流（条件分支、循环、并行节点），对于"单 Agent 多工具"场景来说过度工程化。我们的 Agent Core 在 Vercel AI SDK 的 `maxSteps` + 自定义 event loop 之上就能实现足够的控制力。如果以后需要多 Agent 编排，再引入不迟。

## 五、运行时配置系统

### 5.1 `.mycode/` 目录

项目根目录下的 `.mycode/` 是 Agent 的运行时配置目录，**不提交到 npm 包中**，仅在开发或部署时存在于项目根目录。这与 `packages/`（构建时源码）完全分开。

```
.mycode/
├── mycode.jsonc          # 主配置文件
└── skills/               # 技能文件（Skill）
    └── <skill-name>/
        └── SKILL.md      # YAML frontmatter + Markdown 指令
```

设计理念：
- **配置即代码**：所有运行时可变的配置集中在一个文件，不散落在代码里
- **不污染源码**：`.mycode/` 不参与构建，修改配置不需要重新编译
- **用户可定制**：每个项目可以有自己的 `.mycode/`，内容不同

### 5.2 `mycode.jsonc` 配置结构

配置文件使用 JSONC 格式（支持注释，方便写说明）：

```jsonc
{
  // ── LLM Provider ──
  // defaultProvider 指定当前使用的提供商。
  // 每个 provider 通过 format 字段声明 API 格式：
  //   "openai"    → 使用 @ai-sdk/openai（兼容 DeepSeek / OpenAI）
  //   "anthropic" → 使用 @ai-sdk/anthropic（兼容 DeepSeek Anthropic 端点）
  "llm": {
    "defaultProvider": "deepseek-openai",
    "providers": {
      "deepseek-openai": {
        "format": "openai",
        "baseUrl": "https://api.deepseek.com",
        "apiKey": "",
        "model": "deepseek-v4-flash",
        "contextWindow": 1_000_000,
        "maxOutputTokens": 384_000
      },
      "deepseek-anthropic": {
        "format": "anthropic",
        "baseUrl": "https://api.deepseek.com/anthropic",
        "apiKey": "",
        "model": "deepseek-v4-pro",
        "contextWindow": 1_000_000,
        "maxOutputTokens": 384_000
      }
    }
  },

  // ── MCP 服务器 ──
  "mcpServers": {
    "filesystem": {
      "command": "node",
      "args": ["./mcp-servers/filesystem.mjs"],
      "env": {}
    }
  },

  // ── Agent 运行时 ──
  "agent": {
    "systemPrompt": "You are a helpful coding assistant.",
    "maxSteps": 25,
    "maxContextTokens": 128_000
  },

  // ── 技能系统 ──
  "skills": {
    "enabled": true,
    "paths": [".mycode/skills"]
  },

  // ── 安全策略 ──
  "safety": {
    "toolTimeoutMs": 60_000,
    "maxToolCallsPerMinute": 20,
    "allowedCommands": ["ls", "cat", "find", "grep", "node", "git"]
  }
}
```

### 5.3 配置加载流程

```
Agent 启动
  │
  ├── 1. 查找 .mycode/mycode.jsonc
  │      └── 不存在 → 使用内置默认值
  │
  ├── 2. 读取配置，混合环境变量
  │      ├── mycode.jsonc 中的 apiKey
  │      └── 环境变量 MYCODE_API_KEY 优先
  │
  ├── 3. 根据配置初始化组件
  │      ├── LLM Adapter（选择 provider + model）
  │      ├── MCP Client Manager（连接 mcpServers）
  │      ├── ToolRegistry（注册内置工具 + MCP 工具）
  │      └── Safety 策略（超时 / 限频 / 白名单）
  │
  └── 4. 扫描 skills/ 目录，加载技能
         ├── 扫描默认路径 .mycode/skills/
         ├── 扫描配置中额外路径（skills.paths）
         ├── 按名称去重（先到先得）
         └── 注入到 system prompt
```

### 5.4 技能系统（Skills）

技能是用户自定义的 **文档式指令**，通过 `SKILL.md` 文件编写，Agent 启动时自动发现并注入到 system prompt，让 LLM 在对话中感知和使用。

#### 目录结构

```
.mycode/skills/
└── code-review/
    ├── SKILL.md          # YAML frontmatter + Markdown 正文（必需）
    └── reference.md      # 辅助参考文件（可选，通过相对路径引用）
```

#### SKILL.md 格式

```markdown
---
name: code-review
description: 审查代码质量、发现安全隐患和最佳实践问题
---

# Code Review Skill

审查代码时关注：
1. 错误处理模式
2. 安全风险
3. 可维护性
```

| Frontmatter 字段 | 必需 | 说明 |
|---|---|---|
| `name` | ✅ | Skill 名称，`[a-z0-9]+(-[a-z0-9]+)*`，须与目录名一致 |
| `description` | ✅ | 一句话描述用途，LLM 据此决定何时触发该 skill |

#### 发现与加载

- **默认路径**：`<projectRoot>/.mycode/skills/`
- **额外路径**：通过 `mycode.jsonc` 的 `skills.paths` 配置
- **去重规则**：
  - 扫描路径去重（相同物理路径只扫一次）
  - Skill 名称去重（同名 skill，先发现的保留）
- **加载时机**：Agent 构造函数中执行扫描，结果注入到 system prompt

## 六、核心抽象：Agent Event Stream

这是整个架构中最关键的抽象——Agent Core 与前端之间的唯一通信协议。

### 事件类型定义

```typescript
// packages/core/src/event.ts

/**
 * Agent 产生的所有事件类型。
 * 任何前端（Ink CLI、React Web、SDK 调用方）都通过此类型驱动渲染。
 */
export type AgentEvent =
  // ── 会话生命周期 ──
  | { type: 'session_start'; sessionId: string; timestamp: number }
  | { type: 'session_end'; sessionId: string; reason: string; timestamp: number }

  // ── 思考/推理 ──
  | { type: 'thinking_start'; turnId: string }
  | { type: 'thinking_delta'; turnId: string; delta: string }       // 流式文本块
  | { type: 'thinking_end'; turnId: string; fullText: string }

  // ── 工具调用 ──
  | { type: 'tool_start'; turnId: string; toolName: string; args: unknown }
  | { type: 'tool_delta'; turnId: string; toolName: string; delta: string }  // 工具输出流
  | { type: 'tool_end'; turnId: string; toolName: string; result: unknown }
  | { type: 'tool_error'; turnId: string; toolName: string; error: string }

  // ── 最终回答 ──
  | { type: 'answer_start'; turnId: string }
  | { type: 'answer_delta'; turnId: string; delta: string }
  | { type: 'answer_end'; turnId: string; fullText: string }

  // ── 安全/系统事件 ──
  | { type: 'loop_detected'; turnId: string; toolSequence: string[] }
  | { type: 'context_compressed'; turnId: string; before: number; after: number }
  | { type: 'error'; turnId: string; code: string; message: string };
```

### Agent 主循环签名

```typescript
// packages/core/src/agent.ts

export class Agent {
  constructor(private config: AgentConfig) {}

  /**
   * 启动一个 Agent 轮次。
   * 返回 AsyncGenerator，调用方通过 for await...of 消费事件流。
   */
  async *run(input: string): AsyncGenerator<AgentEvent, void, undefined> {
    // 1. 会话生命周期
    yield { type: 'session_start', sessionId: this.session.id, timestamp: Date.now() };

    // 2. Think → Act → Observe 循环
    //    内部使用 Vercel AI SDK 的 streamText + maxSteps
    for await (const event of this.agentLoop(input)) {
      yield event;
    }

    yield { type: 'session_end', sessionId: this.session.id, reason: 'completed', timestamp: Date.now() };
  }
}
```

### CLI 消费模式

```typescript
// packages/cli/src/hooks/use-agent-stream.ts

async function runAgent(input: string, dispatch: (event: AgentEvent) => void) {
  const agent = new Agent(config);
  for await (const event of agent.run(input)) {
    dispatch(event); // React setState 更新 Ink 渲染
  }
}
```

export async function POST(req: Request) {
  const { input } = await req.json();
  const agent = new Agent(config);

  const stream = new ReadableStream({
    async start(controller) {
      for await (const event of agent.run(input)) {
        controller.enqueue(`data: ${JSON.stringify(event)}\n\n`);
      }
      controller.enqueue('data: [DONE]\n\n');
      controller.close();
    },
  });

  return new Response(stream, {
    headers: { 'Content-Type': 'text/event-stream' },
  });
}

## 七、流式传输架构

### 本地模式（CLI）

```
Agent Core
  │
  AsyncGenerator<AgentEvent>
  │
  ▼
Ink TUI (同一进程内直接消费)
  ├── useAgentStream hook → useState 驱动渲染
  ├── <StreamingText /> — 逐 token 渲染 Markdown
  ├── <ToolCall /> — 工具调用状态卡片
  └── <DiffView /> — 代码差异对比
```

CLI 模式下不需要任何网络传输。Agent Core 和 Ink 共享同一个 Node.js 进程。

### 远程模式（Web）

```
Agent Core
  │
  AsyncGenerator<AgentEvent>
  │
  ▼
Next.js API Route (SSE 端点)
  │ 将 Generator 映射为 data: {...}\n\n
  │
  EventSource (浏览器)
  │
  ▼
React Web UI
  ├── useAgentSSE hook → useState 驱动渲染
  ├── <Chat /> — 对话界面
  ├── <ToolCall /> — 工具调用卡片（样式不同但逻辑同 Ink 版）
  └── <DiffView /> — Monaco Editor 代码对比
```

### 为什么要用 SSE 而不是 WebSocket

| 对比 | SSE | WebSocket |
|------|-----|-----------|
| 单向流（服务器→客户端） | ✅ 原生 | 需要手动实现 |
| 浏览器 API 简洁度 | `EventSource` 一行搞定 | 需要 `new WebSocket()` + 事件绑定 |
| 自动重连 | ✅ 内置 | ❌ 需手动实现 |
| HTTP 兼容 | ✅ 走标准 HTTP | ⚠️ 需要升级握手 |
| 客户端→服务器消息 | 需要额外 POST | ✅ 原生双向 |
| 适合场景 | **Agent 事件推送** | 聊天、实时协作 |

结论：**Agent → 浏览器是典型的服务器推送场景**，SSE 是正确选择。如果需要 Agent 执行期间用户发送中断/取消指令，额外开一个 POST 端点即可，没必要为此引入 WebSocket。

**什么时候该用 WebSocket**：如果 Agent 需要长时间运行（>30 分钟），并且需要双向实时通信（用户可以在 Agent 思考时发送指令），那么 Hono 的 WebSocket 支持可以作为备选方案。

### 内置工具 vs MCP 工具

```typescript
// 内置工具：直接 TypeScript 函数
const readFile = {
  name: 'read_file',
  description: '读取文件内容',
  parameters: z.object({ path: z.string() }),
  execute: async function* ({ path }) {
    yield { type: 'progress', message: `Reading ${path}...` };
    const content = await fs.readFile(path, 'utf-8');
    return { content };
  },
};

// MCP 工具：通过 @modelcontextprotocol/sdk 代理
class MCPTool implements Tool {
  constructor(
    private mcpClient: Client,
    private toolDef: ToolDefinition
  ) {}

  get name() { return this.toolDef.name; }
  get description() { return this.toolDef.description; }
  get parameters() { return this.toolDef.inputSchema; }

  async *execute(args: unknown) {
    // 通过 MCP stdio/HTTP 传输调用远程工具
    const result = await this.mcpClient.request({
      method: 'tools/call',
      params: { name: this.name, arguments: args },
    }, CallToolResultSchema);

    return result;
  }
}
```

### MCP 服务器管理

```typescript
// packages/core/src/tools/mcp/manager.ts

export class MCPClientManager {
  private clients = new Map<string, Client>();

  /** 启动一个本地的 MCP 服务器子进程，连接到 Agent */
  async connectServer(name: string, config: MCPServerConfig) {
    const transport = new StdioClientTransport({
      command: config.command,
      args: config.args,
    });

    const client = new Client({
      name: `my-agent-${name}`,
      version: '1.0.0',
    });

    await client.connect(transport);

    // 获取该服务器提供的所有工具
    const { tools } = await client.request(
      { method: 'tools/list' },
      ListToolsResultSchema
    );

    // 注册为可执行工具
    for (const toolDef of tools) {
      this.registerTool(new MCPTool(client, toolDef));
    }

    this.clients.set(name, client);
  }
}
```

### 工具安全策略

| 安全措施 | 实现 | 说明 |
|----------|------|------|
| **执行超时** | `AbortController` + `Promise.race` | 默认 60s，可配置 |
| **调用频率限制** | Token bucket 算法 | 每工具每分钟 N 次 |
| **命令白名单** | Shell 工具内置 allowlist | 禁止 `rm -rf`、`sudo` 等危险命令 |
| **文件沙箱** | 限制读写范围到工作目录 | 通过路径检测确保不逃逸 |
| **用户确认** | 高风险操作触发 `confirm` 事件 | CLI 弹确认提示，Web 弹对话框 |

## 九、Agent 循环安全与防护

### 已知陷阱及对策

| 风险 | 现象 | 对策 |
|------|------|------|
| **无限工具循环** | Agent 不停地调用工具，从不产生最终回答 | `maxSteps` 硬限制（Vercel AI SDK 内置）；`loop_detected` 事件检测重复工具序列 |
| **Token 耗尽** | 上下文膨胀导致 LLM 调用超时或费用失控 | 上下文压缩策略（摘要历史消息）；Token 计数预警 |
| **LLM 输出异常** | 产生格式错误 JSON，工具解析失败 | Zod 校验 + 重试（最多 2 次） |
| **级联工具失败** | 工具 A 失败，Agent 用错误结果继续调用工具 B | 工具错误传播到 LLM 让模型自行恢复，但连续 3 次失败后终止轮次 |
| **幻觉工具参数** | LLM 捏造不存在的参数 | Zod schema 严格校验；`additionalProperties: false` |

### Agent 循环伪代码

```typescript
// Vercel AI SDK 的 maxSteps 已处理大部分安全逻辑
// 以下是 Agent 需要自己补充的防护层

async *agentLoop(input: string): AsyncGenerator<AgentEvent> {
  let stepCount = 0;
  const maxSteps = 25;          // 硬限制
  const toolSequence: string[] = [];

  // Vercel AI SDK 的 streamText + maxSteps 自动处理多轮 tool call
  const result = streamText({
    model,
    system: this.systemPrompt,
    prompt: input,
    tools: this.toolRegistry.getSchema(),
    maxSteps,                     // ← SDK 内置安全限制
    onStepFinish: ({ toolCalls }) => {
      stepCount++;
      for (const tc of toolCalls) toolSequence.push(tc.toolName);
    },
  });

  // 将 SDK 的 fullStream 事件映射为我们的 AgentEvent
  for await (const chunk of result.fullStream) {
    switch (chunk.type) {
      case 'text-delta':
        yield { type: 'thinking_delta', turnId, delta: chunk.textDelta };
        break;
      case 'tool-call':
        yield { type: 'tool_start', turnId, toolName: chunk.toolName, args: chunk.args };
        break;
      case 'tool-result':
        yield { type: 'tool_end', turnId, toolName: chunk.toolName, result: chunk.result };
        break;
      case 'error':
        yield { type: 'error', turnId, code: 'llm_error', message: chunk.error.message };
        break;
      // ...
    }
  }

  // 检测循环模式
  if (this.detectLoop(toolSequence)) {
    yield { type: 'loop_detected', turnId, toolSequence };
  }
}
```

## 附录：与原始设计讨论的差异说明

以下是对之前讨论中一些假设的修正或细化：

1. **Monorepo 工具**：使用 pnpm workspaces（而非 npm workspaces），因为生态主流项目均采用，且更严格的依赖隔离。
2. **异步模式**：从 "WebSocket or SSE" 明确为 **SSE 优先**，Agent 事件推送是典型的单向流，不需要 WebSocket 的双向开销。
3. **事件传递**：从 "标准的事件流" 具体化为 **`AsyncGenerator<AgentEvent>`**，这一抽象在 Gemini CLI、assistant-ui、Pi 中均得到验证。
4. **Ink 选型**：确认 Ink v7 适合（Claude Code 用它），但需要额外搭配参数解析库（meow/oclif）和自定义流式 hook。
5. **Vercel AI SDK**：从 "`generateText` + `toolCall`" 更新为 **`streamText` + `maxSteps` + `fullStream`** 模式，v6 已原生支持多轮工具调用的自动循环和事件分发。
6. **MCP SDK 版本**：确认使用 **v1.x** 生产，v2 尚在预研阶段（`main` 分支 pre-alpha）。
7. **工具执行**：明确了内置工具和 MCP 工具通过统一 `Tool` 接口融合，而非分开对待。
8. **安全层**：补充了工具超时、频率限制、命令白名单、`loop_detected` 检测等之前未讨论的机制。
9. **添加了 `packages/shared`**：用于 CLI 和 Web 共享 Markdown 渲染、Diff 计算等逻辑，但在原始设计中未提及。
10. **Hono 定位**：从 "Web 后端主力" 调整为 **可选轻量组件**（MCP 服务器、独立 API），主力 Web 框架仍是 Next.js。
